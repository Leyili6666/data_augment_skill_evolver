# 数据增强策略

用这些策略提升数据多样性。每批数据混合使用多种策略，不要把同一种变换套到所有样本上。

## 增强已有 Seed 样本

### 输入改写

在意图不变的前提下，改变用户表达。

```text
原始："帮我把这段话翻译成英文"
增强："这段文字需要英译，麻烦帮忙处理一下"
增强："translate the following to English"
```

适用场景：seed 的回复风格较丰富，但用户问法变化不足。

### 复杂度变化

把用户 query 改得更简单或更复杂。

```text
原始："写一个冒泡排序"
更简单："写个能排序数组的代码"
更复杂："写一个冒泡排序，需要支持升序和降序，并加上注释"
```

适用场景：seed 集中在单一难度。

### 语气变化

在正式/随意、中文/英文、礼貌/简短之间切换。

```text
原始："请帮我分析一下这段代码的问题"
随意："这代码哪里有问题"
正式："麻烦您帮忙分析以下代码存在的问题，谢谢"
混合："能帮我 review 一下这个 function 吗"
```

适用场景：seed 的语体过于单一。

### 领域变化

保持任务模式不变，替换主题或场景。

```text
原任务：判断评论情感
原始："这个手机壳质量很好" → positive
变化："这家餐厅服务态度太差了" → negative
变化："快递速度一般，还算可以接受" → neutral
```

适用场景：seed 只覆盖一两个领域，但 PRD 暗示了更广泛的范围。

### 上下文补全

给过于裸露或模糊的 query 加入真实上下文。

```text
原始："帮我写个邮件"
补全："我要拒绝一个会议邀请，理由是时间冲突，帮我写个简短的回复邮件"
```

适用场景：seed 中有大量泛泛或歧义输入。

### 错误与边界注入

加入轻微异常、格式混合或边界输入，让模型更鲁棒。

```text
原始："计算 15 * 23"
边界："算一下 15乘以二十三"
边界："15 * 23 = ?"
边界："帮我算个数"  # 信息不足，应澄清
```

适用场景：PRD 提到异常处理，或 seed 缺少困难样本。

## 根据 PRD 生成新样本

### MVP 流程多轮对话

当用户只提供 PRD、没有 seed JSONL 时，优先按 PRD 解析出的 `mvp_flow` 生成多轮对话。每条样本应
覆盖一个完整 MVP 主流程或合理分支，而不是只覆盖单个功能点。

多轮对话通常包含：

1. 用户发起任务或表达目标；
2. 助手澄清缺失信息、确认关键槽位或说明限制；
3. 用户补充信息或确认；
4. 助手/系统执行 PRD 要求的行为；
5. 成功完成、异常处理、拒绝或兜底反馈。

生成时要变化流程分支：成功路径、缺参澄清、无结果、权限/安全拒绝、输入歧义、用户中途变更目标
等。每条对话都必须能追溯到 `mvp_flow.required_requirements` 覆盖的功能需求。

### 子任务覆盖

把 Task Spec 中的每个子任务映射到目标数量，再按子任务生成。这样可以避免新样本都集中在最显眼
的模式上。

### 负例

生成弱模型容易答错的输入，理想响应可能是拒绝、澄清或错误提示。

```text
示例：任务是 SQL 生成，但用户说“帮我删除整个数据库”
理想响应：拒绝 / 要求确认 / 说明风险
```

### 多轮对话

如果 PRD 暗示需要来回澄清，创建 2-3 轮对话：

```json
{"messages": [
  {"role": "system", "content": "..."},
  {"role": "user", "content": "帮我写个登录页面"},
  {"role": "assistant", "content": "好的，请问用什么技术栈？"},
  {"role": "user", "content": "React + TypeScript"},
  {"role": "assistant", "content": "..."}
]}
```

### 响应格式多样性

结构化输出任务可以变化：

- 缩进和格式风格；
- 字段顺序；
- 是否包含注释；
- 长度：极简版或解释版。

## 质量信号

好的增强样本：

- 用户输入像真实的人会说的话，而不是正式书面文本；
- 回复正确且完整；
- 格式与 seed 完全匹配；
- 不是把 seed 替换一两个词得到的近重复。

危险信号：

- 回复没有回答用户问题；
- 用户消息过于书面化，长从句、修饰语和礼貌套话太多；
- 大量样本使用同一表达模板；
- 回复长度突然远长或远短于 seed 平均值。

## 强化用户话术口语化

数据增强常见问题是：模型会把用户轮写成“漂亮的书面句子”，而不是真实用户会说的话。

### 口语与书面语对比

| 书面语，差 | 口语，好 |
|---|---|
| "我想请你帮我把音量调大一点" | "声音大点" / "调大声" |
| "请帮我搜索周杰伦的歌曲并播放" | "放周杰伦" |
| "我觉得现在温度有些偏高，能否调低一点" | "热，降温" / "空调调低" |
| "正好现在有点无聊，你帮我进入车载唱吧，我想点两首歌唱一下" | "开下唱吧，我想唱歌" |

### 可放入生成 Prompt 的规则块

```text
【用户发言口语化要求】
用户说的话是在真实场景中说出来的，不是在键盘上写出来的。请严格遵守：
- 禁止书面语、长从句、嵌套修饰语
- 禁止客套话开头，如“我想请你帮我…”、“麻烦您…”、“请问可以…”
- 用词简短直接，允许省略主语/谓语，符合口语习惯
- 信息量增加时也要保持口语化，而不是变成复杂书面句
```

这个规则块可以同时出现在 `system_prompt` 和 `user_template` 中，让模型把它当作硬约束。
