# Prompt 写作模式

创建 `generation_prompt.json` 或 `evaluation_prompt.json` 时读取本文件。这里把通用提示词工程模式改写为
适合数据增强工作流的规则。

## Prompt Spec 组装顺序

按以下顺序组织提示词：

1. 角色与任务：本轮模型要做什么。
2. 不可变契约：输出 schema、合法 JSON、数量要求、禁止额外文本。
3. 领域上下文：PRD 参考、源数据画像、项目 Skill 规则和已批准全局规则。
4. 覆盖计划：功能需求、类别、边界情况、负例和目标分布。
5. 变换规则：什么必须保持不变，什么可以变化。
6. 示例或源记录：few-shot 示例、`{source_record}` 或 `{generation_references}`。
7. 自检：要求模型返回前做简短内部检查，但除非输出契约要求，不要输出推理过程。

稳定、可缓存的内容放在 `system_prompt`；每轮变化的变量放在 `user_template`。

## 结构化输出规则

生成提示词必须要求返回一个 JSON 对象：

```json
{"examples": []}
```

评估提示词必须要求返回一个 JSON 对象，包含数值维度、`issues`、`highlight` 和 `failure_tags`。
如果启用仲裁，还要包含 `verdict`、`confidence` 和 `disagreements`。

明确要求模型只返回合法 JSON，不输出 Markdown code fence、解释性前缀或尾随注释。脚本仍会容忍
code fence，因为模型可能不完全遵守。

## Few-shot 与参考选择

示例是行为锚点，不是可复制内容。

优先选择覆盖以下类型的示例：

- 主流程；
- 一个边界或歧义案例；
- 一个困难或历史失败案例；
- 一个对格式敏感的记录。

源数据很大时，不要把所有记录一次性塞进提示词。改用：

- `reference_files` 插入紧凑 PRD 或规则摘要；
- `generation_mode: "generalize_by_record"` 逐条泛化源记录；
- 少量精选 few-shot，用于锚定格式和语气。

避免示例污染：如果示例的 schema、标签空间、语气或领域与目标契约不同，不要选用，除非提示词明确
说明这种差异的目的。

## 数据增强变换

生成时同时写明不变量和可变项。

常见不变量：

- 输出字段名和 JSON 结构；
- intent、label、function ID 或目标行为；
- 安全边界和拒绝策略；
- assistant 角色和回复完整性。

可变项：

- 表达方式、长度、语体和方言；
- 槽位值、实体或场景上下文；
- 信息顺序；
- 难度；
- 歧义、缺失信息或负向条件。

`generalize_by_record` 模式下，要求模型保持源记录的任务意图和标签语义，同时改变表层表达和场景
细节。明确禁止重复或近重复变体。

## 面向失败模式的提示词约束

定稿 Prompt Spec 前，针对可能失败模式加入可检查约束：

- JSON 无效或缺少必填字段；
- 生成数量不匹配；
- 复制 seed 示例；
- 低多样性、模板化；
- 未覆盖 PRD 需求；
- label 漂移或 intent 漂移；
- 不安全回答、策略不一致或领域能力幻觉；
- 回复过长，破坏期望风格。

如果已有评估证据，把重复失败转成 prompt 约束或示例。不要把孤立失败变成持久项目规则。

## Prompt 质量清单

写入 `generation_prompt.json` 前检查：

- 脚本占位符齐全且拼写正确；
- 输出契约清晰、可解析；
- 覆盖目标可衡量；
- 示例少而准，且格式一致；
- 区分硬要求和偏好；
- 源数据是紧凑参考或逐条插入，不是无控制地整批倾倒；
- 没有 API Key 或私密信息。

写入 `evaluation_prompt.json` 前检查：

- 评估维度匹配用户质量目标；
- 始终包含 `format` 和 `relevance`；
- 评委能识别复制 seed、低多样性和未覆盖 PRD；
- `failure_tags` 是稳定字符串，可供 proposal 使用；
- 仲裁可以比较评委结果，而不是重新生成数据。

## Prompt 优化闭环

用生成批次测试 prompt：

1. 根据确定性校验和评委分数建立基线。
2. 按 failure tag 和需求来源归类失败。
3. 修改能解决重复失败的最小 prompt 片段。
4. 重新生成新批次，不复用相同样本。
5. 除非新增上下文能直接减少失败，否则保持 prompt 更短。

跟踪指标：合法 JSON 率、契约通过率、平均分、评委分歧数、重复/近重复率、功能需求覆盖率和 token
成本。
